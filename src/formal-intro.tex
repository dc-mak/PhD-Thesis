In \cref{sec:c-lang}, I discussed how the \kl{ISO standard} of C attempts to
reconcile the competing forces of portability, hardware control, and
optimisation. I also mentioned that it is ambiguous, and diverges from \kl{de
facto} C. Its rules are challenging to adhere to manually, and complex to
express over C-like languages; existing program logics thus simplify some of
the tricky ones around control flow, variable scoping, aliasing, loose
evaluation order and pointer manipulation.

\kl{CN}'s headline goal (\cref{sec:cn-intro}) is to work with pre-existing C
programs, ones whose correctness may rely on those tricky rules. Yet,
expressing those rules over a C-like language would quickly make it unwieldy,
and thus hamper its other goal of reducing the expertise required for
verification.

So, \kl{CN} builds upon the \kl{Cerberus} executable and empirically validated
semantics for C~\sidecite{memarian2022cerberus}. It thus benefits from both the
\emph{accuracy} and the \emph{usability} of this semantics for both \kl{ISO}
and \kl{de facto} C. This is because \kl{Cerberus} is elaborated into a
relatively small calculus \emph{\kl{Core}}, which translates all of C's
complexity into a first-order functional language with a few special
constructs.

Additionally, \kl{CN} is intended to be used like a \emph{type system} in an
IDE than a program logic in a proof assistant. Ideally, instead of seeing
intermediate goals in a sophisticated separation logic, and being proficient
with a range of inference rules and automation tactics, a user sees their C
program, scattered with predictable and lightweight annotations, in an editor
which either indicates success, or a helpful error message.

% Aside from the fact that the notion and mode of use of a type system is more
% familiar to most programmers, this approach also allows \kl{CN} to use and
% advance the extant literature on building refinement type systems on top of
% existing languages.
%
% This type system approach also leads to other desiderata and their
% corresponding responses.

A good type system aims to minimise obvious annotations and be predictable on
the necessary ones; to do so, \kl{CN} tracks the flow of information
systematically, using a \kl{bidirectional} approach. A type system will also
need automation to not burden the programmer with proving obvious things.
Similar to VeriFast~\sidecite{jacobs2011verifast},
Gillian-C~\sidecite{santos2020gillian}, and
Frama-C~\sidecite{kirchner2015frama}, \kl{CN} enlists SMT solvers to help. A
type system must also terminate, even with SMT solver, so \kl{CN} restricts the
expressiveness of the assertion language and SMT queries. Lastly, to track and
manage resources in C, the type system needs to be substructural.

\kl{CN}'s assertion language aims to be expressive for verifying real-world
programs, but restrictive to ensure termination (and speed), and intuitive for
systems programmers who are familiar with functional programming.

In this part, I formalise and prove type-safe \kl{Kernel CN}: a formalism for
\kl{CN} which assumes all quantifiers are explicitly instantiated, and all
\kl{resource manipulations} have proof terms with \kl{linear types}. It does
not require proof terms for the logical properties (by construction, all
entailments fall are decidable). \kl{Kernel CN} is a \kl{bidirectional} type
system for \intro{ResCore}, a \kl{let-normal} version of \kl{Core}.

I also formalise two related elaboration algorithms: one for quantifier
instantiations, one for \kl{iterated} predicate indices. I do so to explain the
\emph{principles} of \kl{resource inference}, since the implementation
frequently changes; I have not embarked on any proofs for either scheme.

In~\fancyref{chap:formal-background}, I expand upon \kl{CN}'s constraints and
design decisions, explain the relevant features of \kl{Core}, and cover the
necessary background to understand \kl{CN}'s syntax and \kl{Kernel CN}.

In~\fancyref{chap:kernel-grammar}, I explain the \kl{Kernel CN}'s types, and
how they relate to \kl{CN}'s syntax. I also explain the \kl{ResCore} and how it
relates to \kl{Core} by example.

In~\fancyref{chap:kernel-statics}, I explain the \kl{Kernel CN} typing rules
for \kl{ResCore}. It lacks inference and so is simpler to prove sound. Since it
handles the majority of C, the entire system is very large; I will only discuss
the main features. I also show the aforementioned elaboration algorithms, which
infer the information assumed by \kl{Kernel CN}.

In~\fancyref{chap:kernel-soundness}, I discuss the key definitions, lemmas and
proof sketches used in a syntactic proof of type safety for \kl{Kernel CN}. Of
note are the definition of substitution, the novel structure of the heap, and
the interaction between resource terms and the heap.

In~\fancyref{chap:inform-impl}, I discuss how the formalisation brings clarity
to many implementation questions around lifting restrictions or adding new
features.

In~\fancyref{chap:kernel-alternative}, I conclude this part by sketching out an
alternative presentation of the \kl{CN} type system, \kl{MiniCN}, which is
closer to both C as a surface language and the \kl{CN} implementation. It will
demonstrate the challenges of \emph{not} using \kl{ResCore}. It will also
demonstrate the challenges which seem to be inherent to designing a type system
for a C-like language.

\chapter{Background: Cerberus and CN design}%
\label{chap:formal-background}

\margintoc{}

In this chapter, I set out \kl{CN}'s design goals and origins, recapitulate the

\section{\kl{CN} design goals and constraints}%
\label{sec:cn-goals}

Aiming for \emph{``a verification tool whose aspirational goal is to lower the
prerequisites of C verification from a Rocq programmer who knows separation logic to a
systems programmer who is proficient with functional programming''}
(\cref{sec:cn-intro}) helps narrow down the large design space of such
tools.

There were two reasons for picking this goal. First, it was an unexplored area
in the design space. Second, we wanted to verify the pKVM hypervisor, developed
by Google~\cite{deacon2020virtualization} (and its precursor, Hafnium).

To explain \kl{pKVM}, I need to explain its context: the Android operating
system. Android runs on billions of devices worldwide and so handles an
enormous amount of sensitive data. Security updates are paramount, but each
device runs its own kernel: up to half the code is \emph{not} Android's version
of the Linux kernel. This makes testing and deploying updates, challenging,
expensive and slow: it takes more than 18 months for new releases of Linux and
to roll-out to devices. Aside from security, fragmentation and fewer
upstreamed features and fixes, are also concerns.

% \url{https://youtu.be/7novnkldMmQ}

Whilst some of these concerns have been mitigated with the introduction of
Generic Kernel Images
(GKI),\sidenote{\url{https://source.android.com/docs/core/architecture/kernel/generic-kernel-image}}
which provide a small and stable kernel ABI \emph{for a particular long-term
release} of Android, there are still fundamental issues with this model. Both
the kernel (20 million lines of code), and device drivers shipped by vendors,
are too large to be a reasonable trusted computing base.

Some vendors try to use hypervisors to isolate the kernel from the rest of the
system \textemdash{} such as `secure' hardware storing sensitive data
\textemdash{} by running Android and other hardware components in separate
virtual machines. They also use hypervisors for non-security reasons, such as
partitioning memory at boot-time, direct memory access, or arbitrary code
execution (at a more privileged level) outside of Android. Using hypervisors
just \emph{shifts} the attack
surface,\sidenote{\url{https://googleprojectzero.blogspot.com/2020/02/mitigations-are-attack-surface-too.html}}
and also increases fragmentation.

Similar to \kl{GKI}, the proposed solution is to standardise the hypervisor.
There is already a mature hypervisor inside the Linux kernel: the Kernel-based
Virtual Machine (KVM)\@. It lets a host kernel dynamically allocate virtual
machines for guests, and protects the host from them. However, initially, it
did not protect guests from the \emph{host}. This is a problem because the
guest could be running code for a secure hardware component (e.g.\ a biometric
authenticator), but the host could be a compromised version of Android, so an
attacker could still get access to sensitive information.

To solve this, Google, as part of the Android Virtualisation
Framework,\sidenote{\url{https://source.android.com/docs/core/virtualization/whyavf}}
is developing a \intro[pKVM]{protected KVM
(pKVM)},\sidenote{\url{https://lpc.events/event/7/contributions/780/}} which
runs \emph{underneath} the kernel,\sidenote{pKVM utilises the Armv8 exception
    levels (EL) feature, which defines four levels of increasing privilege. EL0
    is used for Android applications, EL1 is used for the kernel, EL2 is used
for the hypervisor, and, EL3 is not used.} and ships \emph{as part} of the
kernel image. This allows \kl{pKVM} to remain very small, only supporting a few
functions (such as virtual memory management), and rely on the Linux kernel to
manage the rest, such as scheduling, device drivers and power management.

If successful, this would make the attack surface much smaller in code, but
much larger in number of devices. It is in this context that my research group
and Google saw the opportunity to verify a small hypervisor, \emph{on an
ongoing basis}. A one-and-done verification which takes an army of PhD students
and postdocs a few years to verify is not worth the investment; a tool which C
kernel programmers can understand, use and maintain security-critical proofs
with, would be.

This background explains \kl{CN}'s headline goal, and also
clarifies some of the \emph{constraints} on its design:
\begin{itemize}
    \item Because kernel programmers wrote the code, and are intending to use
        conventional compilers to build and run it, we cannot hope
        some \adhoc{} model of C is sound \textemdash{} we want and need
        something that matches and can handle C's real-world behaviour as
        closely as possible.
    \item Because it will be used by kernel programmers, we want a story that
        is acceptable to them. They are very smart and capable people, who do
        not have the time or support to get up to speed with separation logic
        and interactive proof assistants. Nor are they amenable to changing
        their development or company's product delivery workflows
        substantially. I conjecture that a ``fancier type system'' which runs
        as part of their continuous integration (CI) pipeline has a better
        chance of being adopted.
    \item Similarly, because the annotations will be read by kernel
        programmers, and upstreamed into Linux, we want them to be minimal and
        easy to understand. Achieving this requires careful management
        about the flow of information in the type system, and automating
        as many obvious things as feasible.
\end{itemize}

In turn, these constraints feed into concrete technical choices which \kl{CN}
makes:
\begin{itemize}
    \item To capture real-word C behaviour, \kl{CN} uses the \kl{Cerberus}
        empirically validated semantics.
    \item To integrate into existing workflows, \kl{CN} appears to users
        as a fancy type system.
    \item To retrofit the type system on top of the existing one, and in
	particular to ensure erasure, \kl{CN}'s type system uses
	\kl{refinement types}.
    \item To automate proofs of constraints, \kl{CN} relies on SMT solvers.
    \item To ensure decidability (termination), and aim for reasonable in
        practice performance, \kl{CN} follows a \emph{liquid} typing
	discipline, restricting the queries it sends to the SMT solver.
    \item To minimise constraint annotations, \kl{CN}'s type system is
        formalised and implemented in a \kl{bidirectional} style.
    \item To check the resource management of C programs, \kl{CN}'s type
        system uses \kl{linear} types, using the grammar of separation
        logic assertions.
    \item To stay within the \kl{liquid} typing discipline \emph{and} avoid
        quantifier instantiation annotations, \kl{CN} uses precise assertions
        with a \kl{mode}d discipline for predicate arguments.
    \item To make such precise assertions with a mode discipline easier to
        read, write and explain, \kl{CN} uses a \emph{monadic}
        syntax.
\end{itemize}

\section{Cerberus and Core for a usable and accurate C semantics}%
\label{sec:cerberus-core}

\intro{Cerberus} is an empirically validated and executable semantics for
\kl{ISO} and \kl{de facto} C, specifically C11. A detailed comparison
between it and other C semantics is available in the Related Work chapter
of \sidetextcite{memarian2022cerberus}, but for the purposes of \kl{CN},
it suffices to say it captures real-world C.

\kl{Cerberus} is vital for \kl{CN} because it expresses this executable
semantics in a \emph{relatively simple first-order functional language},
\kl{Core}, via a \kl{compositional elaboration}.

This drastically simplifies many tricky parts of C. \kl{Cerberus} handles
the strange scoping and control flow (of jumping \emph{into} a loop) of
\cref{fig:perplexing-ub}. It parses, and statically checks the program.
However, it declares the program illegal during execution because of
subtle, dynamically checked rules (see caption). Whilst contrived, any C
function which happens to have exception handling and retry behaviour, and be
longer than the height of a programmer's screen, could make the same error.
\kl{Cerberus} handles C features such as mutable and potentially aliasing
variables, and the myriad other \kl{undefined} or \kl{unspecified} behaviours,
\eg{} loose evaluation order and coercions respectively, by making them
all \emph{explicit} in \kl{Core}'s syntax.

\section{Core grammar}\label{sec:core-grammar}

\kl{Cerberus} elaborates C into \intro{Core}, a not-too-large calculus where
each construct is designed to capture some peculiarity of C. It is split into
two fragments, a pure fragment (\cref{fig:pure-core-grammar}) and an effectful
on (\cref{fig:effectful-core-grammar}); the former is embedded into the latter.

The pure fragment forbids memory operations, but
does include the effect of undefined behaviour explicitly with the
\coreinline{undef()} operator. It handles % chktex 36
things like implicit type conversions or bounded arithmetic. As visible from the
grammar, the pure part is very much a first-order functional language with
recursive functions, with some C-specific constructs such as pointer
arithmetic on arrays/members and specified/unspecified/integer values.

The effectful fragment supports memory operations, sequencing constraints,
and exotic control flow with a goto-like operator used for elaborating
C's iteration and \cinline{goto} statements. The distinction between the pure
and effectful fragments is unrelated to the distinction between expressions and
statements in C:\ both are elaborated into effectful expressions (for example,
\coreinline{PtrEq} which tests for pointer equality).

I will discuss \coreinline{memop()} in more detail in % chktex 36
\fancyref{chap:mem-model-explained}. For now it suffices to say that the
operations are effects, part of the memory interface \kl{Core} uses to abstract
over different memory object models.

Next are the constructs related to sequencing. A more precise treatment
of this is given later in \fancyref{sec:weak-seq}. As per
\sidetextcite{memarian2022cerberus}, \emph{Each use of an action is either
``positive'' (the default case) or ``negative'' (if the action appears in the
syntax as the operand of the $\cnkw{neg()}$ operator)}.
\begin{itemize}
    \item $\cnkw{neg}(\cnnt{action})$ marks side
        effects (see text to note~\ref{sn:side-effect}).
    \item $\cnkw{unseq}(E_1, \ldots, E_n)$ is unsequenced evaluation of each
        sub-expression.
    \item $\cnkw{let}\ \cnkw{strong}\ \cnnt{pat}\ \cnkw{=}\ \cnnt{E}_1
        \ \cnkw{in}\ \cnnt{E}_2$ sequences all memory actions performed by
        $\cnnt{E}_1$ before those performed by $\cnnt{E}_2$.
    \item $\cnkw{let}\ \cnkw{weak}\ \cnnt{pat}\ \cnkw{=}\ \cnnt{E}_1
        \ \cnkw{in}\ \cnnt{E}_2$ only only sequences positive memory actions of
        $\cnnt{E}_1$ before those performed by $\cnnt{E}_2$. In other words,
        any negative memory actions ($\cnkw{neg}(\cnnt{action})$) performed by
        $\cnnt{E}_1$ are unsequenced with respect to all memory actions
        performed by $\cnnt{E}_2$. A race only occurs if unsequenced memory
        actions have overlapping footprints (\cref{sec:weak-seq}), and is
        deemed \kl{UB}.
    \item $\cnkw{bound}(\cnnt{E})$ evaluates its argument $\cnnt{E}$ and
        then clears its footprint (\cref{sec:weak-seq}). It marks the
        boundary of a C expression as a statement.
\end{itemize}

The \coreinline{par()} construct is for spawning threads. The % chktex 36
\coreinline{nd()} construct models non-determinism as required or % chktex 36
implied by aspects of the standard.

The \coreinline{ccall()} and \coreinline{pcall()} constructs are for % chktex 36
calling elaborated C functions and \kl{Core} procedures (effectful functions)
respectively. They differ only in how the name of the procedure to be called is
found, with \coreinline{ccall()} using the memory interface to do so. % chktex 36

The (thread-local) Core operational semantics is defined in
\textcite{memarian2022cerberus}; I present the key rules below.
It is a small-step transition from a configuration to either undefined
behaviour or a new configuration: $\langle h, E, \kappa
\rangle \rightarrow (\textsc{Undef} \mid{} \langle h', E', \kappa' \rangle)$,
for a heap $h$, effectful expression $E$, and stack $\kappa$ (list of
evaluation contexts $C$). These steps rely on evaluating pure expressions (in a
big-step style) to either undefined behaviour or a value $\cnnt{pce} \Downarrow
(\textsc{Undef} \mid{} \cnnt{value})$.

For example, \coreinline{pcall()} pushes the enclosing evaluation context $C$ % chktex 36
on to the stack $\kappa$, evaluates its arguments into values, looks up the
function body $E$, substitutes the parameters for the values, and then executes
the body.

{\small%
\[
\inferrule[{[PCall]}]
 { \mathrm{funmap} ( \cnnt{name} ) = \left(\cncomp{x_i {:} \beta_i}{i}\right).\ E \\
   \cncomp{\cnnt{pce}_i \Downarrow \cnnt{value}_i}{i} }
 { \left< h , C\left[ \cnkw{pcall} \, \left( \cnnt{name}, \cncomp{\cnnt{pce}_i}{i} \right) \right] , \kappa \right>
   \rightarrow \left< h , \left[ \cncomp{\cnnt{value}_i / x_i}{i} \right] E , C \Colon \kappa \right> }
\]}

Conversely, \coreinline{return()} pops an evaluation context off of the stack, % chktex 36
applies it to the returned value, and proceeds with the resulting expression.

{\small%
\[
\inferrule[{[Return]}]
  { }
  { \langle \cnnt{h} , \cnkw{pure}(\cnnt{value}) , C \Colon \kappa \rangle
    \rightarrow \langle h, C [\cnkw{pure}(\cnnt{value})] , \kappa \rangle }
\]}

The \coreinline{save()} operator is \kl{Core}'s way of introducing named % chktex 36
continuations with default arguments. This is the aforementioned \cinline{goto}-like
operator, which is used to elaborate C's iteration and
\cinline{goto} statements.
\begin{itemize}
    \item $\cnnt{id}$ is the name, also referred to as the label.
    \item $x_1, \ldots, x_n$ are parameters which in scope in $E$.
    \itemAP{} $e_1, \ldots, e_n$ are \intro{default arguments} for those parameters.
\end{itemize}

\textbf{The label of \coreinline{save()} expression is scoped to its % chktex 36
enclosing \kl{Core} procedure.} This means there are three ways of reaching
one.

\begin{itemize}
    \item Forward jump, with a \coreinline{run()} operator. % chktex 36
    \item Backwards jump, with a \coreinline{run()} operator. % chktex 36
    \item No jump, via evaluation of the enclosing expression.
\end{itemize}

A \coreinline{save()} expression evaluates by instantiating % chktex 36
and substituting its parameters in its body. The instantiation
depends on how the expression was reached. If it was via the
\coreinline{run()} operator, then the instantiation is % chktex 36
with the arguments of the same. If it was via the enclosing expression,
then the instantiation is with the \kl{default arguments}.

Formally, \coreinline{save()} (reached normally, via its enclosing % chktex 36
expression) evaluates its default arguments into values and then substitutes
those values into its body.\sidenote{Since this does not
manipulate the heap or the stack, it is phrased purely as expression
reduction, with a separate, omitted rule to lift expression reductions to
thread-local reductions.}

{\small%
\[
\inferrule[{[Save]}]
  { \cncomp{\cnnt{pce}_i \Downarrow{} \cnnt{value}_i}{i} }
  { \cnkw{save} \, \cnnt{id} \left( \cncomp{x_i \cnkw{:=} \cnnt{pce}_i}{i} \right) \, \cnkw{in} \, \cnnt{E}
    \rightsquigarrow \left[ \cncomp{ \cnnt{value}_i / x_i }{i} \right] E }
\]}

Reaching the same \coreinline{save()} via a \coreinline{run()}  % chktex 36
is more complicated. Like a \coreinline{pcall()}, it evaluates its arguments % chktex 36
into values and looks up the label $\cnnt{id}$. The $\mathrm{labelmap}$ is
precomputed on a per-procedure basis by \kl{Cerberus} during elaboration.
Unlike \coreinline{pcall()}, the result of the lookup is \emph{a context} % chktex 36
$C_{\cnnt{id}}$ and an expression $E_{\cnnt{id}}$. It substitutes the values
for the parameters in $E_\cnnt{id}$, \emph{discards the current context} $C$,
and uses $C_\cnnt{id}$ instead. All the while, the stack $\kappa$ is unchanged.

{\small%
\[
\inferrule[{[Run]}]
  { \mathrm{labelmap} ( \cnnt{id} ) = \left(\cncomp{x_i}{i}\right).\ C_{\cnnt{id}} [ E_{\cnnt{id}} ] \\
    \cncomp{\cnnt{pce}_i \Downarrow \cnnt{value}_i}{i} }
  { \langle h , C\left[ \cnkw{run}\, \cnnt{id} \left( \cncomp{\cnnt{pce}_i}{i} \right) \right] , \kappa \rangle
    \rightarrow \left< h , C_{\cnnt{id}} \left[ \left[ \cncomp{\cnnt{value}_i / x_i}{i} \right] E_{\cnnt{id}} \right] , \kappa \right> }
\]}

\textbf{For our purposes, the most important thing is that while control flows
into \coreinline{ccall()}, \coreinline{pcall()} and \coreinline{run()}, it % chktex 36
only returns from the first two and not the last.}

\subsection{Discussion}\label{subsec:elab-discussion}

A few things are noteworthy about the Core elaboration:
\begin{itemize}
    \item The elaboration is \intro{compositional}: each function, block,
        statement and expression is elaborated in isolation, based only
        on its parts, and follows the structure of the original \kl{C} program.
    \item Each variable (function argument, local, and global) is uniformly
        handled as living on the heap and gets its own storage via the
        \coreinline{create} function. Reads, writes, and de-allocations are
        represented with \coreinline{load}, \coreinline{store},
        \coreinline{kill} respectively.
    \item Loose evaluation order (for example between the expressions of a \cinline{==})
        is represented using \coreinline{unseq} and \coreinline{let weak} constructs.
    \item \kl{UB} is made explicit in the syntax of the program.
\end{itemize}

\kl{Compositionality} is just as important, as the \kl{Core} being a
first-order functional language. This makes it straightforward transport
annotations on C source, via the elaboration, to annotations on \kl{Core}.

If \kl{Cerberus} elaborated C into a dataflow graph instead,
such transporting might be difficult. Function pre- and postconditions would
be easy, but how to handle loop invariants, \coreinline{goto} annotations, and
\kl{CN statements (proof hints)} is unclear.

In principle, the \kl{compositional} mapping also ensures that errors in
verifying \kl{Core} can be related back to useful locations in the original C.
In practice, this requires a good amount of engineering to accomplish
(\cref{sec:error-msgs}).

Another challenge is distance between C and \kl{Core}. Reporting SMT assertion
failures in terms of what the user wrote is difficult when it mentions
variables introduced by the elaboration, or when the assertion itself is
introduced by \kl{CN} (\cref{sec:counter-ex}).

This concludes the background related to \kl{Cerberus}; I will now discuss the
background related to the \kl{CN} type system.

\section{Decidable refinements for retrofitting and counter-examples}\label{sec:decidable-refinements}

\kl{Core} already has a straightforward \kl{bidirectional} type system which
\kl{Cerberus} uses as a sanity check for the \kl{elaboration}. It has not been
used for a proof of soundness, which would just be a sanity check on the
dynamic semantics.

\begin{marginfigure}
    \centering
    \begin{mathpar}
        \inferrule{ }{\Gamma{} \vdash{} \cnkw{undef} (\cnnt{UB\_name})\Leftarrow{} \cnnt{T}}
    \end{mathpar}
    \caption{Checking rule for the \coreinline{undef()} pure expression as % chktex 36
        mentioned in \textcite{memarian2022cerberus}.}\label{fig:core-ub-typing}
\end{marginfigure}

For example, \coreinline{undef()} checks against any type % chktex 36
(\cref{fig:core-ub-typing}). This is because \kl{UB} terminates the dynamic
semantics; it does not return control. Remember, a valid C program is one which
has no \kl{UB}, but such a type system cannot guarantee its absence.

To do that, we need to enrich the context to track the (control flow) path
conditions. Then, the typing rule for \coreinline{undef()} would % chktex 36
require that the context be inconsistent (able to prove false), meaning
the expression is unreachable. To illustrate this, consider a trivial example
of nested \mintinline{py}{if} statements in \cref{fig:dead-code}.

\begin{figure}[h]
    \inputminted[linenos,numbersep=6pt,mathescape,fontsize=\small]{py}{code/dead_code.py}
    \caption{A contrived example on how tracking control flow assumptions
        within a program could be used to prove the impossibility of
        undesirable behaviour.}\label{fig:dead-code}
\end{figure}

From the outer \mintinline{py}{if}, we know that \mintinline{py}{x >= 5}, so
the check \mintinline{py}{x < 4} will always evaluate to \mintinline{py}{False}.
Thus control will always flow to line 12 (and return) instead of line 9 (and
raise the exception). Tracking path conditions in a constraint context during
type checking means that the context is inconsistent at line 9.

To track such constraints in a typing context, we need to talk about program
variables and values \emph{at the type level}. This requires richer types which
can depend on program terms. However, we \emph{do not} want the converse; we
should be able to recover the original program and dynamic semantics by erasing
the types after type checking. Enriching the type of an existing language
whilst retaining erasure is known as \intro{refinement
types}~\sidecite{jhala2021refinement}.

When using such types, we do not want the user to provide proofs for obvious
deductions, such as $x \geq 5 \wedge x < 4 \Rightarrow \bot$. Unfortunately,
first-order logic with arithmetic is undecidable: type checking may not
terminate on certain programs. Given that there are real-world programming
languages\sidenote{\href{https://hacklang.org/}{Hack},
\href{https://forums.swift.org/t/swift-type-checking-is-undecidable/39024/34}{Swift},
and F$^*$~\cite{swamy2016dependent}.} which have undecidable type systems, but
work fine in practice, is this a problem?

Whilst `decidable' does include exponential or NP-complete time complexity, we
conjecture that such SMT theories are faster, and more \emph{stable}: the
performance is consistent across irrelevant or small changes to a proof. We
also conjecture that problems generated by type checking will be limited by
human cognitive capacity and thus be reasonably small, especially compared to the
typical uses of SMT solvers. Thus \kl{CN} aims for decidability as an imperfect
proxy.

\kl{CN} ensures decidability by restricting its use of quantifiers as per the
liquid types~\sidecite{rondon2008liquid} approach. In practice, this means
constraints sent to the SMT solver are free of quantifiers. When this too
restrictive, \kl{CN} offers a lemma mechanism (\cref{sec:lemma-prover}) to
export and prove facts in the Rocq~\sidecite{CDT2024} proof assistant.

Aiming for decidability also lets \kl{CN} extract concrete
\kl{counter-examples} to explain constraint failures in terms of concrete
values for \kl{Core} program variables (a subset of which correspond to C
program variables).

\section{Bidirectionality for taming subtyping}\label{sec:bidir-subtyping}

With \kl{liquid types}, we can use an SMT solver to automate tedious proofs.
However, implications induce a subtyping relation in such a type system, which
means we have a \emph{subsumption rule}, as shown in~\cref{fig:subsumption}. As
presented, the rule is \emph{not} syntax-directed, and so raises the question:
when should it be used?

\begin{marginfigure}
  \begin{mathpar}
      \inferrule{\Gamma{} \vdash{} e \mathrel{{:}} \{\, x \mid{} \phi (x) \,\}  \\ \phi (x) \Rightarrow{} \phi' (x) }
                {\Gamma{} \vdash{} e \mathrel{{:}} \{\, x \mid{} \phi' (x) \,\}}
  \end{mathpar}
  \caption{Subsumption rule using for a system with subset types and logical
      implication $\Rightarrow$ as the subtyping relation.}\label{fig:subsumption}
\end{marginfigure}

Revisiting~\cref{fig:dead-code}, imagine a type system where contexts
$\Gamma$ are either $\cdot$ or $\Gamma, \{\, x \mid \phi (x) \,\}$, and
$\phi$ represents some constraint on $x$. In particular, we are not
accumulating \kl{path conditions} in the context, just the variables and their
subset types. We would like to prove that function \mintinline{py}{gt_zero}
satisfies its postcondition, $ \{\, \mathsf{ret} \mid \mathsf{ret} > 0 \,\} $
without raising an exception.

Working forwards, we start with a type $\{\, \mathsf{x} \mid \top
\,\}$. Inside the function, we are checking both branches because we cannot
statically rule one out. In the true case, we \emph{refine} the type to
$\{\, \mathsf{x} \mid \mathsf{x} \geq 5 \,\}$. The subsumption rule is not
syntax-directed, so we are free to apply it now. Doing so, we have have $\{\,
\mathsf{x} \mid \mathsf{x} > 0 \,\}$, because $\mathsf{x} \geq 5 \Rightarrow
\mathsf{x} > 0$. Unfortunately this means we cannot rule out the
\cinline{else}-branch and the exception within.

Whilst quite contrived, it illustrates the importance of knowing when and how
to use the subsumption rule. We could ask users to annotate the code at each C
statement, but this would be extremely burdensome: we would have traded the
tedium of proving simple facts for the tedium of writing obvious annotations.

\kl{CN} chooses a middle-ground: it requires annotations on functions and
loops, as a minimum. Socially, we believe this is easy to explain and accept,
since programmers already write documentation for the former. And technically,
function and loop annotations enable \emph{compositional verification}.
Verifying a large code base can be parallelised and fast, because it can
sidestep the need to do complex and potentially slow loop-unrolling or
inter-procedural analyses.

Concretely, this means we would like \emph{check} types at function calls,
function returns, loop entry, and loop exit, from the top down, and
\emph{synthesise} types everywhere else, from the bottom up. This is exactly
what \kl{bidirectional} type systems were designed to do (Sections 4.4 and 4.6
in~\sidetextcite{dunfield2021bidir}). Moreover, it gives a clear principle of
when to apply the subsumption rule (call the SMT solver): at the boundary
between synthesis and checking (\cref{fig:simple-bidir}).

\section{Linearity to manage (non-leaky) resources}\label{sec:linearity-manage-resources}

Our aim is define a type system rich enough to prove the absence of \kl{UB},
which can arise from both the pure and effectful fragments of \kl{Core}
(\cref{sec:core-grammar}). Whilst decidable refinements and bidirectionality
are suitable for preventing \kl{UB} from the former, we need more techniques to
accomplish the same for the latter, ones capable of handling features such as
such as loose evaluation order or a myriad of subtle memory violations.

In \fancyref{sec:sep-logic-intro}, I sketched out how separation logic can be
used to reason about the behaviour of imperative programs which manipulate a
heap and can have syntactically distinct terms \intro{alias} the same location
in said heap. If we want to embed separation logic propositions into a type
system, then we need to be able to precisely control how those propositions are
handled.

In a typical (intuitionistic) type system, assumptions in the context are
duplicable, for example, the type of a variable in \kl{Core}, or constraints
about pure values. This relies on the fact that $A \Rightarrow A \wedge A$ in
intuitionistic logic. However, the same is not true for $\ast{}$ in separation
logic: $A \nRightarrow A \ast{} A$. This is because assertions have their
semantics defined with reference to a heap. For example, a heap satisfies $A_1
\ast{} A_2$ iff it can be split into two disjoint sub-heaps which satisfy $A_1$
and $A_2$ separately. If $A = (x \mapsto{} y)$, then there is one heap
satisfying $A$, but none for $A \ast{} A$.

Such duplication is known as \intro{contraction} and we need to forbid this to
reason soundly (whilst keeping other assumptions, such as the type of \kl{Core}
program variables, and constraints, contractible).

\begin{marginfigure}
  \begin{mathpar}
      \inferrule[{[Weakening]}]{
          \Delta{}, A \vdash{} C
      }{
          \Delta{} \vdash{} C}
      \\
      \inferrule[{[Contraction]}]{
          \Delta{}, A, A \vdash{} C
      }{
          \Delta{}, A \vdash{} C}
      \\
      \inferrule[{[Exchange]}]{
          \Delta_1, B, A, \Delta_2 \vdash{} C
      }{
          \Delta_1, A, B, \Delta_2 \vdash{} C}
    \\
  \end{mathpar}
  \caption{Substructural sequent calculus rules.}\label{fig:substructural}
\end{marginfigure}

In an intuitionistic setting, we can freely forget things we know.
In a separation logic setting, this depends on whether the user cares about
memory leaks or not (i.e\. the C standard does not make memory leaks an error).
The intended use of \kl{CN} in resource constrained settings
leans us towards being explicit about when we deallocate and more importantly
\emph{when we forget to}.\sidenote{Though not yet supported by \kl{CN},
concurrency also requires being explicit about destroying a resource, for
example, to prevent a user from acquiring the same mutex twice.}
Forgetting an assumption is known as \intro{weakening}, and we would like to
forbid this \emph{only} for heap-related facts; it is sound to forget type
information and ambient constraints.

We would however like the ability to commute assumptions freely, since it is
perfectly acceptable to free in any order, regardless of the order of
allocation, so we keep \intro{exchange}. Sequent calculus rules for contraction,
weakening, and exchange are shown in \cref{fig:substructural}. Jettisoning
weakening and contraction but keeping exchange means that we handle our
(heap-related) separation logic assumptions
\emph{linearly}~\sidecite{girard1987linear} in \kl{CN}.

Note that this is in constrast to popular approaches such as the drop trait in
Rust,\sidenote{\url{https://doc.rust-lang.org/std/ops/trait.Drop.html}} or RAII
(Resource Acquisition Is Initialisation) in
C++,\sidenote{\url{https://isocpp.github.io/CppCoreGuidelines/CppCoreGuidelines\#Re-raii}}.
These allow the programmer to elide explicit resource deallocation, by
automatically inserting such a call when the resources goes out of scope.
Modelling that would require \kl{affine} rather than linear resources.

For completeness, a \kl{relevant} logic is one which supports contraction and
exchange but not weakening. In practice this means relevant assumptions may not
be implicitly dropped, but may be used one or more time.

Linearity may be necessary, but it can also be cumbersome. If a linear resource
represents the permission to dereference a pointer, a typing rule which
consumes that must also return an identical resource, so that it may be
dereferenced again. If we require the programmer to handle these permissions
explicitly in the annotations, such a discipline would quickly violate the goal
to minimise obvious annotations. This means that in addition to automation for
proving constraints, by using SMT solvers, we need to have some scheme to
\intro[resource inference]{infer resources} (see \cref{sec:elaboration}) to
reduce the burden of annotations.

\begin{marginfigure}
  \begin{mathpar}
      %\inferrule[Declarative Var]{
      %}{
      %    x {:} t \vdash{} x {:} t}
      %\\
      %\inferrule[Algorithmic Var]{
      %}{
      %    \Gamma{} , x {:} t \vdash{}  x {:} t \dashv{} \Gamma'}
      %\\
      %\{\, x {:} t \,\} = \Gamma' - \Gamma
      %\\

      \\

          \\

      }{

      \\
      \Delta_1 = \Gamma_2 - \Gamma_1 \and \Delta_2 = \Gamma_3 - \Gamma_2
  \end{mathpar}

\end{marginfigure}

I should note that, for the purposes of the formalisation of \kl{Kernel CN}, I
do use explicit linear resource terms, since that makes the system easier to
state and prove sound. For clarity and convenience, the linear context in the

linear resource terms also make it easy to specify and explain the \kl{CN}
\kl{resource inference} algorithms as an \emph{elaboration}, which I do in

the elaboration rules.} Those rules use the linear context in an algorithmic
manner.

\section{Precise assertions for inferring resources and quantifiers without
backtracking}\label{sec:precise-assertion-inferring}

Lurking behind the decision to use linear separation logic and liquid types is
a tension between the rich scheme of quantifiers needed to write specifications
expressive enough to prove \kl{UB}-freedom and full functional correctness and
the desire to (a) not send quantified formulas to the SMT solver and (b)
minimise the annotations (quantifier instantiations) a user needs to write.

So aside from minimising annotations for proofs of obvious statements (by
carefully using an SMT solver) and minimising annotations for linear resource
terms (by inferring them), we would also, as far as possible, like to
infer how quantifiers are instantiated, based on what the user wrote.\sidenote{
Note that this is not at all related to \emph{inferring specifications}, which
is a large and challenging problem in its own right.} Unfortunately, such
inference is undecidable.

To better understand this problem, let us recall the separation logic proof
sketch of linked integer list append from
\cref{fig:append-annot,fig:list-pred}, reproduced in
\cref{fig:append-annot-formal,fig:list-pred-formal} for convenience.

\begin{marginfigure}
    \small%
    \centering
    \begin{align*}
        \mathrm{list} &(\mathsf{p}, l) \mathrel{{=}^\mathrm{def}} \\
                      &\mathsf{emp} \ast{} (\mathsf{p} = \mathsf{NULL} \wedge{} l = []) \\
                      &\vee{} \exists{} \; {head}, \; {tl}, \mathsf{p\_tail}.\\
                      &\qquad (\mathsf{p} \mapsto{} {head}) \\
                      &\qquad \ast{} (\mathsf{p} + 1 \mapsto{} \mathsf{p\_tail}) \\
                      &\qquad \ast{} \mathrm{list} (\mathsf{p\_tail}, {tl}) \\
                      &\qquad \ast{} l = {head} {:}{:} {tl}
    \end{align*}%
    \caption{Definition of a recursive list predicate in a simple separation
        logic.}\label{fig:list-pred-formal}
\end{marginfigure}

We see that in addition to the pointers $\mathsf{xs}$ and $\mathsf{ys}$, which
can be thought of as a quantification over \intro{computational} variables, the
specification for append also quantifies over \emph{logical} or ghost variables
$l_1$ and $l_2$. Given that the ``computational terms'' are just code, the
computational variables need not be inferred, since the user needs to provide
them for the call to the function to be valid.\sidenote{The same principle
applies for instantiating existentials used in postconditions too. I will
discuss quantifiers related to array reasoning in \cref{sec:it-array}.}

\begin{figure}[h]
    \inputminted[breaklines,mathescape,fontsize=\small]{py}{code/append_annot.py}
    \caption{A separation logic proof sketch of a linked integer list
        append.}\label{fig:append-annot-formal}
\end{figure}

However, it would be a shame if the user had to provide explicit instantiations
for $l_1$ and $l_2$ at each call site, including the recursive call within the
implementation of \mintinline{py}{append}. If we want to stick to a liquid
typing discipline, where every quantifier must be matched up to a program
variable, we need a way to bind the ghost values which were existentially
quantified over in the $\mathrm{list}$ predicate of
\cref{fig:list-pred-formal}, to names the user would have to choose.

This is doubly irksome because prior to the recursive function
call, we already know (a) $\mathrm{list}(\mathsf{xs}', l_1')$ and
$\mathrm{list}(\mathsf{ys}', l_2')$ and (b) the precondition requires choices
of $l_1$ and $l_2$ such that $\mathrm{list}(\mathsf{xs}', l_1)$ and
$\mathrm{list}(\mathsf{ys}', l_1)$ so there is \emph{only one sensible choice}
for those quantifiers.

To recap, when calling a function, we need some way of guessing some
instantiation of quantifiers such that it satisfies the precondition. \emph{If}
we are in a situation where there is only one sensible instantiation given the
required and the available assertions, a simple inference scheme would be to
simply scan the context for predicates which match on constructor (i.e.\
$\mathsf{emp}$ matches with $\mathsf{emp}$, $\_ \mapsto{} \_$ matches $\_
\mapsto{} \_$, $\mathrm{list(\_, \_)}$ matches $\mathrm{list}(\_, \_)$) and
\emph{computational} arguments, and instantiate any quantifiers based on the
remaining values.

So in this instance, when calling \mintinline{py}{append(xs->tail, ys)}, % chktex 36
such an inference scheme would
\begin{enumerate}
    \item Delay instantiating $l_1$ and $l_2$.
    \item Note that the precondition requires $\mathrm{list}(\mathsf{xs}',
        l_1)$ and $\mathrm{list}(\mathsf{ys}', l_2)$.
    \item Check the context for a match with $\mathrm{list}(\mathsf{xs}',
        \_)$.\sidenote{Checking whether two symbolic terms are equal can be
        automated with an SMT solver.}
    \item Find $\mathrm{list}(\mathsf{xs}', l_1')$.
    \item Select $[l_1' / l_1]$ as its instantiation.
    \item Repeat similarly for $\mathsf{ys}$ and $l_2'$.
\end{enumerate}

This is a cute idea, but it glosses over several details, such as
\begin{enumerate}
    \item Are such assertions characterisable, and if so, how?
    \item Can such a scheme handle disjunction, $ P \vee{} Q$?
    \item Are assertions where \kl{computational} arguments uniquely determine
        \kl{logical} arguments, expressive enough for realistic code?
    \item What is the correct atomic predicate to look up in a context?
    \item Can all assertions be decomposed into a context of such atomic
        predicates?
    \item Should it unfold recursive predicates, and if so, when?
\end{enumerate}

I will tackle the first three questions here, and leave the latter three
to be discussed in~\fancyref{sec:heap-types}.

For the first question, the answer is: in practice, \emph{yes}. If one assigns
input and output \kl{modes} to the arguments of separation logic predicates,
and ensures that predicates are \emph{mode-correct} via a syntactic check
(\fancyref{sec:monadic-syntax}), then that suffices to guarantee that the system
can \emph{always infer quantifiers}.

However, the theoretical justification for this unclear. Such assertions happen
to be characterisable as \kl{precise}~\sidecite{reynolds2008intro}, but the
connection to modes has not been formally explained.

\subsection{Precise assertions}\label{subsec:precise-assertion}

\begin{definition}[Precise assertions]%
\label{def:precise-assertion}
    \AP{} An assertion $Q$ is \intro{precise} iff, for all stores $s$, and heaps
    $h$, there is at most one $h' \subseteq{} h$ such that $h' \in [\![ Q ]\!] (s)$.

    In other words, if a \kl{precise} assertion holds on a subheap, then it
    does so uniquely.
\end{definition}

Based on this definition, one can derive some examples and properties of
precise assertions as shown in~\cref{fig:precise} (examples of imprecise
assertions are shown in~\cref{fig:imprecise}). As I shall show later, each
production in the grammar for \kl{linear} \kl{resource types}
(\cref{fig:kernel-res}), stays within the precise fragment.

\begin{marginfigure}
\small%
\begin{mathpar}
    \mathsf{emp}
    \and p \mapsto v
    \and \exists v.\ p \mapsto v
    \and \mathrm{list}(\mathsf{p}, l)
    \and \exists l.\ \mathrm{list}(\mathsf{p}, l)
    \and P \wedge Q \text{ when $P$ or $Q$ is precise}
    \and P \ast Q \text{ when $P$ and $Q$ are precise}
\end{mathpar}%
\caption{Some examples and properties of precise assertions.}\label{fig:precise}
\end{marginfigure}

\citeauthor{brotherston2016model}~\sidecite{brotherston2016model}~showed that
precise assertions and an ordered-disjunction allow for efficient (polynomial time)
model checking without backtracking (since a normal disjunction is enough to
require backtracking and thus exponential).
% in~\fancyref{sec:elaboration}, I
% show how the same allow for \kl{CN} to infer instantiations of quantifiers
% without backtracking. I do so by leaning on input and output \kl{modes} which
% (a) is intuitive, given that in C source a pointer is an ``input'' and
% dereferenced value an ``output'' and (b) happens to coincide.

This brings us nicely to the second question about disjunctions. As visible
from \cref{fig:imprecise}, we see that arbitrary disjunctions pose a problem.
Yet, from \cref{fig:precise}, we see that the list predicate, and even a
version of it which is existentially quantified over its ghost list, fits the
definition of precise, despite including a disjunction in its definition
(\cref{fig:list-pred-formal}).

\begin{marginfigure}
\small%
\begin{mathpar}
    \mathsf{true}
    \and \mathsf{emp} \vee{} \mathsf{x} \mapsto{} 42
    \and \mathsf{x} \mapsto{} 3 \vee{} \mathsf{y} \mapsto{} 7
    \and \exists \mathsf{p}.\ \mathsf{p} \mapsto{} 1
    \and \exists \mathsf{p}.\ \mathrm{list}(\mathsf{p}, l).
\end{mathpar}
\caption{Examples of im\kl{precise} assertions.}\label{fig:imprecise}
\end{marginfigure}

This is because the two arms of the disjunction make assertions
which are disjoint on \emph{both} the values the computational parameter
$\mathsf{p}$ can take \emph{and} on the shape of the heap; one branch says
$\mathsf{p} = \mathsf{NULL}$ and the heap will be $\mathsf{empty}$, whereas the
other branch guarantees that (implicitly) $\mathsf{p} \neq \mathsf{NULL}$ and
that the heap will have \emph{at least two} heap cells. I will return to the
discussion of disjunctions in precise predicates imminently; for now I want to
note that the locations of those two cells are expressed entirely in terms of
program variables.

At this point, a keen reader may have noticed that while this may be true at
the `top-level' of the predicate, the recursive call does use the existentially
quantified ghost value $\mathsf{p\_tail}$, which is on the right of $\mathsf{p}
+ 1 \mapsto{} \mathsf{p\_tail}$. So why is this definition still \kl{precise}?
It is because in any heap which satisfies the non-empty branch of the list
predicate, $\mathsf{p\_tail}$ is \emph{uniquely} determined by its relation as
the value in the location adjacent to $\mathsf{p}$.\sidenote{A proof of this is
left as an exercise to the reader.} This can be operationalised with a \kl{mode}
discipline for predicate arguments, which I shall discuss in
\fancyref{sec:monadic-syntax}.

Finally, the third question, of whether precise assertions are expressive
enough for realistic code is an empirical one. Our experience using \kl{CN} so
far~\sidecite{pulte2023cn,pulte2024tutorial} suggests the answer is yes, it is
sufficient for small to medium tutorial examples, all the way up to specifying
the buddy allocator (\cref{chap:buddy}) used in pKVM\@.

That being said, we have found that being able introduce and instantiate
quantifiers manually can sometimes be useful for ergonomics. An example came
from the efforts of Cassia Torczon to verify a basic string manipulation
library in \kl{CN}.%
\sidenote{\href{https://github.com/rems-project/cerberus/issues/540}{Cerberus\#540}}

The representation of strings in this library are null-terminated arrays of
characters. Because the length is unknown, and the direction of traversal
fixed, null-terminated strings are natural to describe using a recursive
predicate. However, the library functions are implemented using while-loops and
array-indexing, for which it is convenient (but not necessary) for proof (and
library clients) to express strings as an array (represented by an iterated
predicate), over a finite range. The end of that range is the string's length,
which is not available in the code, especially before traversing the string.
However, to prove all indexing is within bounds, we need to be able to refer to
the string's length in the precondition and loop-invariants. Therefore, being
able to introduce and instantiate the string's length manually makes the proof
more convenient.

\section{Monadic syntax for mode-correctness}\label{sec:monadic-syntax}

\intro[mode]{Modes} help us ensure that we can infer quantifiers instantiations
as follows. We start with separation logic predicates, and label their
arguments as either $\mathsf{in}$puts or $\mathsf{out}$puts, as shown in
\cref{fig:mode-pred}. In the scope of each quantifier, for each predicate, we
check the mode of each parameter matches the mode of each argument.
Computational arguments the user provides are considered inputs. If a
quantified variable is used in an input position, it must also be used
elsewhere in an output position.

\begin{marginfigure}
    \centering
    \small%
    \begin{align*}
        \_ \mapsto{} \_ \:&: \mathrm{Loc}^{\mathsf{in}} \times \mathrm{Value}^{\mathsf{out}} \rightarrow \mathrm{Prop} \\
        \mathrm{list} \:&: \mathrm{Loc}^{\mathsf{in}} \times \mathrm{IntList}^{\mathsf{out}} \rightarrow \mathrm{Prop}
    \end{align*}
    \caption{Modes on separation logic predicates.}\label{fig:mode-pred}
\end{marginfigure}

So for example, the second disjunct of \cref{fig:list-pred-formal}, the
quantified variable $\mathsf{p\_tail}$, occurs in $\mathrm{list}
(\mathsf{p\_tail}, {tl})$ in the input position, but this is fine because it also
occurs in the output position in $\mathsf{p} + 1 \mapsto{} \mathsf{p\_tail}$.
However, in the example from \cref{fig:imprecise}, $\exists \mathsf{xs}. \
\mathrm{list}(\mathsf{xs},l)$ is badly moded because $\mathsf{xs}$ occurs in an
input position, but nowhere else in an output.

Ensuring that quantifiers preserve \kl[precise]{precision} in assertions by
dividing up predicate arguments into input and output modes is
not new~\sidecite{somogyi1996execution,berdine2006smallfoot,nguyen2008runtime,maksimovic2021gillian},
including for the purposes of reducing the burden of
annotation~\sidecite{jacobs2011verifast}.

What is new is the target audience of kernel programmers, rather than
verification specialists. Programmers may not be used to thinking in terms of
ghost state, and discussions with our users suggest that adding quantifiers and
\kl{mode}s on top of that would be step too far.

An elegant solution to both inscrutable mode rules, and the problem of
disjunctions mentioned earlier, is to change the perspective on
\cref{fig:mode-pred}~\sidecite{krishnaswami2022monadic}. A predicate
$\mathrm{list} : \mathrm{Loc} \times \mathrm{IntList} \rightarrow
\mathrm{Prop}$, considered set-theoretically is just $\mathcal{P} (\mathrm{Loc}
\times \mathrm{IntList})$, which is isomorphic to $\mathrm{Loc} \rightarrow
\mathcal{P} (\mathrm{IntList})$. Unlike the initial form, which gave ``equal
weighting'' to input and outputs, this form makes the input role of
$\mathrm{Loc}$ and the output role of $\mathrm{IntList}$, inside a covariant
powerset monad, very natural.

\begin{marginfigure}
    \small%
    \begin{align*}
        &\llbracket \_ \rrbracket \::\: \tau \rightarrow (\tau \rightarrow \mathsf{Prop}) \\
        &\llbracket \mathsf{return}\ t \rrbracket = \lambda a.\ t = a \wedge{} \mathsf{emp} \\
        &\llbracket \mathsf{Own(t)} \rrbracket = \lambda a.\ t \mapsto{} a \\
        &\llbracket \mathsf{let\ x} = e; e' \rrbracket \\
        &\: = \lambda a.\ \exists x.\ \llbracket e \rrbracket x \ast{} \llbracket e' \rrbracket a \\
        &\llbracket \mathsf{if}\ t\ \mathsf{then}\ e\ \mathsf{else}\ e' \rrbracket \\
        &\: = \lambda a.\  \left( t \wedge \llbracket e \rrbracket a \right) \vee \left( \neg \; t \wedge \llbracket e' \rrbracket a \right)
    \end{align*}
    \caption{Monadic syntax for separation logic, along with a translation into the traditional presentations. Pure
        terms are denoted by $t$, and monadic expression are denoted with $e$.}\label{fig:monad-sl}
\end{marginfigure}

This notational transformation turns \emph{mode correctness} into
\emph{variable scoping}, when one interprets let-bindings $\mathsf{let\ x} = e;
e'$ as monadic bindings inside a covariant powerset monad. It also enforces the
disjointness of computational values and the shape of the heap mentioned
earlier with the $\mathsf{if}\ t\ \mathsf{then}\ e\ \mathsf{else}\ e'$
construct. Not only can all logical quantifiers now be \emph{inferred} without
backtracking, branching assertions can be checked without backtracking too:
check $e$ if you can can prove $t$, or check $e'$ if you can prove $\neg\;t$.

\section{\kl[iterated]{Iterated} separating conjunctions to handle arrays}\label{sec:it-array}

C programmers use arrays a lot, particularly with computed index (i.e.\
`random') access, rather than following a particular traversal pattern to get
some element in the middle.

Typically (such as in VeriFast) arrays are handled via recursive predicates,
and if the order of traversal in C does not match the order of traversal in the
predicate, or access is required to the middle of the array, then a user needs
to use lemmas of some sort to massage the available assertions into a usable
form.

\kl{CN} implements \intro{iterated separating conjunctions}, inspired by
\sidetextcite{muller2016automatic}, but with restrictions to ensure only
quantifier-free SMT queries. They are introduced and eliminated as follows.
\begin{itemize}
\item $ \cnkw{each}  \left(  \beta \,  \cnmv{x} \, ;  \cnnt{iguard}
    \right)\, \left\{  { \alpha }  \left(  \cnkw{array\_shift} \, \langle
    \tau  \rangle  \left(  { y_p }  \cnsym{,}  \cnmv{x}  \right)  \cnsym{,}
    \cnnt{iargs}  \right)  \, \right\} $. % chktex 25
\begin{itemize}
    \item $\beta$ is the type of the quantifier (usually % chktex 25
        \cninline{u64}).
    \item $\cnmv{x}$ is the iterated variable. \item $\cnnt{iguard}$ is the index
        guard, a boolean-valued logical/SMT expression on the iterated variable
        $\cnmv{x}$.
    \item ${ \alpha }$ is the name of a predicate.
    \item $\cnkw{array\_shift} \, \langle  \tau  \rangle  \left(  { y_p }  \cnsym{,}  \cnmv{x}  \right)$ is \kl{CN} syntax for C % chktex 25
        pointer-arithmetic ${ y_p } \,  \mathbin{ \symtt{+} }  \, \cnmv{x}$ where the base pointer ${ y_p }$
        has C type $\tau  \cnsym{*}$. This requires the first argument of any
        ${ \alpha }$ to be a pointer.
    \item $\cnnt{iargs}$ is a list of logical/SMT arguments.
\end{itemize}
\item Indices which are intended to move into or out of an iterated separating
    conjunction must be declared so explicitly with a $\cnkw{focus}\ \cnnt{term}\cnkw{;}$
    statement.
\end{itemize}

This enables a few convenient features.
\begin{itemize}
    \item Iterated separating conjunctions can be tested for equality by
        looking only at the name of the predicate being iterated over, symbolic
        equality of the base pointer, and symbolic equivalence of the guard
        conditions.
    \item Predicates can be moved out of and into iterated separating conjunctions
        automatically.
    \item Iterated separating conjunctions can be split along an arbitrary
        guard automatically in a similar fashion, except `subtraction' is done as follows:
        for guard \cninline{A = 0i32 <= i && i < 10i32} and guard
        \cninline[breaklines]{B = A && mod(i, 2i32) == 0i32}, % chktex 36
        the subtraction is simply \cninline{A && !B}. Note that the split % chktex 26
        does not have to be contiguous.
\end{itemize}

It also comes with a few limitations.
\begin{itemize}
    \item It does not support automatically \emph{merging} arrays, because the
        constraints required to express the \emph{merged values} are outside
        the decidable fragment for the SMT theory of arrays.\sidenote{\kl{CN}
        could, but does not, support merging for uninitialised arrays.}
    \item It does not make it clear what the lower and upper bounds of the
        iteration is. This would be useful for both disjointness reasoning in
        proof mode, and runtime checking for
        Fulminate~\sidecite{banerjee2025fulminate}.
\end{itemize}

\kl{CN} did support the decidable array property
fragment~\sidecite{bradley2006whats}, but it was not expressive enough for
verifying pKVM's \kl{buddy allocator} (\cref{chap:buddy}) and so it was
removed. \kl{CN} also supported a scheme for inferring indices (instead of
requiring the movable ones to be declared up front), but this was removed
during a change to the inference
scheme.\sidenote{\href{https://github.com/rems-project/cerberus/commit/7c2c0a364a4373e4eb109f32d01cc9584f51e81f}{Commit
7c2c0a36.}}\label{sn:new-inf}
